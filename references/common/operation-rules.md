# 옷차원 공통 운영 규칙

## git push 자동 실행 절대 금지

```
모든 에이전트는 git push를 자동으로 실행하지 않는다.
push 전 반드시 아래 내용을 사용자에게 보고하고 허가를 받는다.

[push 허가 요청 형식]
━━━━━━━━━━━━━━━━━━━━
Push 허가 요청
- 대상 브랜치 : feature/product-3d-viewer
- 커밋 수     : 3개
- 변경 파일   : ThreeViewer.tsx, product.service.ts, ...
- 실행 명령어 : git push origin feature/product-3d-viewer
━━━━━━━━━━━━━━━━━━━━
push를 진행해도 될까요?
```

## 커밋 메시지 컨벤션 (Conventional Commits)

```
<type>: <한글 또는 영어 설명>

※ scope 표기 금지. 예: `feat(3d): ...` 형태 사용하지 않음.

type:
  feat      새 기능 추가
  fix       버그 수정
  docs      문서 수정
  style     코드 포맷 (기능 변경 없음)
  refactor  리팩터링
  test      테스트 코드 추가/수정
  chore     빌드/설정 변경
  perf      성능 개선

예시:
  feat: 상품 상세 3D 뷰어 컴포넌트 추가
  fix: JWT 토큰 만료 시 무한 갱신 루프 수정
  test: FittingService 단위 테스트 추가
  refactor: 주문 도메인 레이어 분리
```

## 브랜치 전략

| 브랜치 | 용도 | 규칙 |
|--------|------|------|
| `main` | 릴리즈 브랜치 | 직접 push 금지, PR만 허용 |
| `develop` | 통합 개발 브랜치 | PR로만 머지 |
| `feature/*` | 기능 개발 | develop으로 PR |
| `fix/*` | 버그 수정 | develop으로 PR |
| `hotfix/*` | 긴급 패치 | main + develop 동시 PR |

## PR 규칙

```
1. PR 대상 브랜치: develop (main 직접 PR 금지)
2. PR 제목: 커밋 컨벤션과 동일 형식
3. 변경 파일 5개 초과 시 PR을 기능 단위로 분리
4. 테스트 통과 필수 (테스트 없는 PR 금지)
5. PR 머지는 반드시 사용자가 직접 수행
```

## AI 에이전트 작업 시 주의사항

1. **git push 자동 실행 금지** — 반드시 사용자 허가 후 실행 (위 형식으로 보고)
2. 커밋 전 반드시 `git status`로 변경 파일 확인
3. 민감 정보(토큰, 비밀번호 등)는 `.env` 파일에 분리하고 `.gitignore`에 등록
4. `main` 브랜치 직접 push 절대 금지 — PR을 통해 병합
5. 커밋 메시지는 Conventional Commits 형식 준수
6. 작업 전 항상 최신 코드를 `git pull`로 동기화
7. 테스트 코드 없는 기능 구현 금지 (FIRST 패턴 준수)

## 운영 플로우 (PM 선보고 필수)

### 1) PM 선보고
- PM은 매 일차 시작 전에 사용자에게 먼저 보고한다.
  - 오늘 목표(완료 기준 포함)
  - Sub Agent별 핵심 작업 1~3개
  - 예상 산출물 경로/형태

### 2) Sub Agent 실행
- PM 선보고 이후에만 Sub Agent가 작업을 시작한다.
- 작업 중간 보고는 최소화하고, 완료 시점에 결과 중심으로 보고한다.

### 3) 일차 종료 보고
- PM은 Sub Agent 결과를 수집해 사용자에게 한 번에 보고한다.
- 보고 형식: "가장 중요한 작업 1개 + 작업물 경로"

### 4) Git 규칙
- commit: Sub Agent 자율 수행
- push: 반드시 사용자 승인 후 실행

### 5) 보고서 규칙
- placeholder 문구 금지: 예) `(repo 내 ... 반영)`, `TBD`
- 보고에는 반드시 아래 2개 포함:
  - 가장 중요한 작업 1개(구체 내용)
  - 실제 작업물 경로(절대경로/Figma 프레임명/URL)

### 6) MEMORY.md 업데이트 규칙 (필수)
- **작업 종료 시 반드시 `MEMORY.md`를 업데이트**한 뒤 커밋/종료한다.
- MEMORY.md 위치: **프로젝트 폴더 하위 루트** (`ot-chawon/MEMORY.md`) — 다른 경로 절대 금지.
- 업데이트 항목:
  - 완료된 작업 체크박스 (`[ ]` → `[x]`) + commit hash
  - 새로 확인된 패턴/규칙
  - 변경된 주요 파일 경로
- MEMORY.md 업데이트 없이 커밋/종료 금지.

## Sub Agent 간 협업 매트릭스

| 발신 → 수신 | 전달물 / 협의 항목 |
|-------------|-------------------|
| **PM** → 전체 | 스프린트 목표, 작업 배정, 우선순위, 완료기준(DoD) |
| **UI/UX** → FE | 디자인 handoff 통보, 프레임 스펙 |
| **UI/UX** → AI/3D | AI 피팅 결과 포맷, 3D 뷰어 시각화 레이아웃 |
| **FE** → BE | API 응답 포맷 협의, Swagger 명세 기준 연동 |
| **BE** → FE | API 명세(OpenAPI) 공유, 응답 구조 변경 사전 통보 |
| **BE** → DevOps | 서비스별 환경변수 목록, 포트/의존성 |
| **AI/ML** → 3D | 피팅 AI용 리깅 스펙, SMPL 파라미터 포맷 |
| **AI/ML** → BE | 피팅 추론 API 스펙, 입출력 포맷 |
| **3D** → FE | GLB 에셋 전달, Three.js 렌더러 호환 포맷 확인 |
| **3D** → AI/ML | 리깅된 아바타/의류 3D 모델 납품 |
| **QA** → 전체 | 버그 리포트(심각도 분류 기준), 테스트 결과 |
| **DevOps** → 전체 | 인프라 URL, CDN 경로, 환경변수 제공 |

## 민감 데이터 처리 공통 원칙

1. **최소 수집 원칙** — 기능 구현에 필요한 최소한의 데이터만 수집한다.
2. **암호화 저장** — 비밀번호(BCrypt), 신체정보(AES-256), 결제정보(PCI-DSS 준수).
3. **코드 내 하드코딩 금지** — 토큰, 비밀번호, API 키는 `.env` 또는 시크릿 관리 도구에 분리.
4. **테스트 데이터에 실 개인정보 사용 금지** — 픽스처/시드 데이터는 익명 더미 데이터만 사용.
5. **AI 피팅 결과에 실 사용자 사진/신체 노출 금지** — 3D 아바타 기반으로 처리.
6. **접근 권한 최소화** — 서비스/인프라 접근은 역할에 필요한 최소 권한만 부여.

## 버그 심각도 분류 체계 (모든 Sub Agent 공통)

| 레벨 | 기준 | 목표 처리 시간 |
|------|------|--------------|
| **Critical** | 서비스 전체 불가, 데이터 손실, 결제 오류 | 즉시 (당일 핫픽스) |
| **High** | 핵심 기능 불가 (3D 뷰어/피팅 장애), 대다수 사용자 영향 | 다음 스프린트 최우선 |
| **Medium** | 기능 일부 제한, 우회 방법 존재 | 현재 스프린트 내 |
| **Low** | UI 미세 오류, 사용에 지장 없음 | 백로그 등록 후 여유 시 |

## Jira 기반 작업 운영 규칙 (모든 Sub Agent 공통)

> Jira는 모든 작업의 단일 진실원천(Single Source of Truth)이다.

### 기본 원칙

1. **티켓 없는 작업 금지** — 모든 작업은 Jira 티켓이 존재해야 시작할 수 있다.
2. **티켓 상태 전환 필수** — 작업 시작/완료 시 반드시 티켓 상태를 전환한다.
3. **커밋과 티켓 연결** — 커밋 메시지에 티켓 번호를 포함한다. 예: `feat: 상품 3D 뷰어 추가 (OTC-123)`
4. **완료 기준은 티켓 DoD** — 티켓에 명시된 완료기준을 충족해야 Done 전환 가능.

### 티켓 상태 전환 규칙

```
To Do → In Progress → In Review → Done
                 ↘ Blocked (블로커 발생 시)
```

### 핸드오프 규칙

1. **산출물 완료 시** — 후속 에이전트의 티켓에 코멘트로 산출물 경로/스펙을 남긴다.
2. **Blocked 해소 시** — Blocked → In Progress로 전환하고 코멘트를 남긴다.
3. **핸드오프 코멘트 필수 항목**: 산출물 경로, 변경 사항 요약, 주의사항.
